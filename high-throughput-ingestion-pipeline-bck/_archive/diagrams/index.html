Architecture Highlights

  • Mode: Batch Processing (glueetl) - NOT streaming
  • Batching Strategy: File-count based (100 files per manifest)
  • Concurrency: 30 Glue jobs running in parallel
  • Workers: 20 × G.2X workers per job (16GB RAM each)
  • Event Flow: S3 → SQS → Lambda → DynamoDB → Manifest → Glue
  • Monitoring: CloudWatch + DynamoDB metrics + Custom dashboards

Key Design Features

Batch Processing

Uses AWS Glue batch mode (glueetl) instead of streaming for better cost control and resource management. Jobs are triggered per manifest.

File-Count Batching

Groups exactly 100 files per manifest regardless of total size. Optimal for 3.5-4.5 GB files with 500GB safety limit.

Parallel Processing

30 Glue jobs run concurrently, each with 20 workers. Processes 3,000 files simultaneously for 8-15 hour daily completion.

State Tracking

DynamoDB tracks file status through pending → manifested → completed states. GSI enables efficient querying of pending files.

Error Handling

Invalid files quarantined based on size validation. Failed jobs logged to CloudWatch with retry capability.

Compression

Snappy compression achieves 5-7x reduction. 1.3 PB NDJSON → 190-270 TB Parquet daily, saving storage costs.
